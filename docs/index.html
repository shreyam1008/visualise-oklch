Why OKLCH

A more usable color model for UI systems, design tokens, and Tailwind-era workflows.

Perceptual lightness

Adjusting the first channel changes perceived brightness more predictably than traditional HSL-style models, which makes token systems easier to tune and reason about.

Chroma that behaves

Saturation shifts feel more intentional, which is why OKLCH is showing up in Tailwind, design systems, theme editors, and modern CSS tooling.

Modern CSS-native syntax

The syntax is readable, composable, and already available in current browsers and editor ecosystems that track modern CSS color specs.

Built for editor performance

Minimal extension-host work, even when files get large.

Shared decoration pools

The extension reuses one decoration type per rendered swatch instead of creating a fresh decoration for every match. That cuts allocation churn where the old implementation paid the highest cost.

Auto scan strategy

Smaller documents scan in full for correctness. Larger documents switch to padded visible-range scanning so the extension does not waste time decorating code you cannot currently see.

Small package footprint

The published VSIX contains only the bundled extension, manifest metadata, and project essentials. The current packaged artifact stays under 55 KB.

Verified toolchain

This fork is tested with `bun test`, typechecked with both `tsc` and native-preview `tsgo`, and bundled into a single runtime file for fast startup.
